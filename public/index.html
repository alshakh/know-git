Repository

  • A Project
  • Self contained
  • Contents are related
  • Shared History 

                        $ git init

Status

  • Get status of repository at current time 

                        $ git status

Commits 1st view

  • Locked State of the Repository
  • Referenced by Unique SHA1 Hash
  • Parent Commit Reference (hashes)

                        $ git add <files/directories>
                        $ git commit

Log

View Commit History


                        $ git log
                        $ git log --patch
                        $ git log --stat
                        $ git log --pretty=oneline
                        $ git log --graph

Why is git the most loyal tool?

Because of it's COMMITment!

Diff

  • Compare two states of repository 

                        $ git diff
                        $ git diff --staged
                        $ git diff <commit 1> <commit 2>
                        $ git diff --stat
                        $ ...

Tags

  • Named refence to a specific commit
  • Can be used instead of commit hash as a reference
  • Usually used to indicate a release version

                        $ git tag <Tag name>

Revert

  • Reverse a certain commit
  • Without deleting the old commit
  • History Does NOT change!
  • New commit

                        $ git revert

Checkout (1)

  • Turn the working are to a specific commit

                        $ git checkout <Commit Hash>

GitIgnore

  • Ignore files from repository repository
  • Any changes to these files are not noticed
  • Filename is .gitignore

Collaboration Model

  • Every one Pushes & Pulls to/from a centra repository

Push

Sharing your commits to the outside

Local previous hsitory and remote history MUST match

Commits 2nd view

  • Just a set of defined changes to repository
  • Almost like an operation
  • Can be applied at any order

Branches 1st view

  • Timeline for the repository
  • Can be forked at any commit
  • You checkout branches to develop

                        $ git branch
                        $ git branch <branch name>

HEAD

HEAD always exists and refers to the current commit

HEAD is usually used with relative references

Checkout (2)

  • Switch to different branch

                        $ git checkout <branch name>

Branches 2nd view

  • Just a reference to a commit (like a tag)
  • It advances everytime a new commit is made
  • A branch is really cheap!

                        $ git branch <branch name> <commit ref>

Reset

Set branch reference to commit


                    $ git reset <reference>
                    $ git reset <reference> --hard

Merging

  • Add commits (changes) from a branch to another branch
  • Two types :
    • Fast forward
    • Merge commit

Keep convensions

  • Always create a README file in every project, agree on format & content
  • Agree on a main branch, i.e the clean stable branch
  • Agree on commit messages format and content
  • Agree on a versioning scheme
  • Agree on a workflow
    • https://www.atlassian.com/git/tutorials/comparing-workflows
      • Centralized Workflow
      • Feature Branch Workflow
      • Gitflow Workflow
      • Forking Workflow
    • Make your own

Secret Information

Keep Secret Information outside the repository

Use .gitignore to ignore credential files

Effictive Use of branches

  • Not every branch should be shared
  • NEVER commit directly to the main branch
  • Always tryout it things in a new branch
  • Keep a main branch with only clean commits
  • Do NOT work on the stable branch, when it's ready merge and push
  • Delete old branches
